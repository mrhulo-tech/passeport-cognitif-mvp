# ALL IN Live Demo Runbook

## 1. Objectif
Disposer d’un déroulé simple, stable et exécutable en direct pour présenter le Passeport Cognitif MVP sans hésitation et sans dépendre d’improvisations.

## 2. Préparation avant démonstration
- Vérifier que le déploiement Vercel est en statut Ready
- Vérifier que le backend Render répond
- Ouvrir à l’avance les URLs utiles
- Préparer un navigateur avec seulement les onglets nécessaires
- Vérifier l’affichage sur un écran externe si présentation publique

## 3. URLs à ouvrir avant la prise de parole
- Frontend principal
- Frontend utilisateur 1
- Frontend utilisateur 2
- Backend Render `/dashboard/1`
- Backend Render `/dashboard/2`

## 4. Ordre recommandé de la démonstration

### Étape 1 - Introduction
Présenter le problème :
les données d’apprentissage existent déjà, mais elles sont souvent peu lisibles, peu actionnables et peu utiles à la personnalisation.

### Étape 2 - Proposition de valeur
Présenter le Passeport Cognitif comme un module LMS qui transforme des données d’apprentissage en un profil apprenant dynamique, lisible et actionnable.

### Étape 3 - Positionnement éthique
Dire explicitement :
ce système ne mesure pas l’intelligence.
Il présente des données observables, des indicateurs interprétables et des recommandations pédagogiques.

### Étape 4 - Vue utilisateur 1
Montrer :
- le profil apprenant
- les métriques principales
- les indicateurs interprétables
- les recommandations

### Étape 5 - Vue utilisateur 2
Basculer vers `?user=2` et montrer que :
- le profil change
- les indicateurs changent
- les recommandations changent
- la logique de personnalisation est visible

### Étape 6 - Message de valeur
Conclure que le LMS ne sert plus seulement à stocker ou afficher des traces, mais à guider une progression et une action pédagogique concrète.

## 5. Messages à répéter pendant la démo
- profil apprenant dynamique
- données observables
- indicateurs interprétables
- recommandations pédagogiques personnalisées
- aucune mesure objective de l’intelligence
- valeur démontrable dans le LMS

## 6. Plan B si un problème survient
### Si le frontend ralentit
Rafraîchir une seule fois puis utiliser la seconde URL déjà ouverte.

### Si le backend ralentit
Montrer la structure déjà chargée et recentrer la présentation sur l’architecture et la logique produit.

### Si une URL échoue
Utiliser directement l’autre profil utilisateur déjà ouvert dans un onglet séparé.

## 7. Rôle du démonstrateur
Le démonstrateur doit :
- garder un rythme calme
- ne pas trop expliquer la technique
- revenir toujours à la valeur pédagogique
- insister sur la transparence et l’utilité concrète

## 8. Résultat attendu
À la fin de la démonstration, l’audience doit comprendre :
- ce qu’est le module
- ce qu’il montre
- à quoi il sert
- pourquoi il est crédible
- pourquoi il est utile immédiatement
